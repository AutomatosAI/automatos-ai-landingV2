§ 01 — North Star

Principles

Five rules. Every design decision should pass all five.

01 · Premium, not playful

Serif headlines, mono details, cream-gold accents

The product is a regulated B2B platform for licensed operators. Every surface should feel like a premium console — not a consumer app. Lean on the serif display face for moments of importance; the mono treatment for technical truth.

02 · Dark by default — always

No more cream-background pages

Every admin page renders on --bs-bg (#07090A). The smoky variant --bs-bg-smoke is reserved for hero sections and modal scrims. The cream/parchment background on Analytics is retired.

03 · One green. One gold. Two roles.

Green = action. Gold = identity.

Brand green (#52D97A) signals affordance and success — primary buttons, "in stock", success badges, focus rings. Cream-gold (#D9BC82 family) signals identity and labels — eyebrows, metric numbers, secondary badges. Never swap roles.

04 · Hairline, not heavy

1px borders, soft shadows, calm hover

Density comes from typography and spacing, not from heavy strokes or saturated fills. Card borders are 1px hairlines, shadows are diffused and downward, hover adds a soft green ring — never a solid colour shift.

05 · Two page rhythms — pick the right one

Centered serif header for "settings & landing" — compact left header for "data & lists"

Settings, dashboards and overview pages use the centered eyebrow + serif headline + descriptive subtitle. Heavy data-table pages (Audit Logs, Customers when filtered, etc) use a compact left-aligned header so the table can breathe. Both are documented in Page Templates.

§ 06 — Do & Don't

Migration corrections

Specific drift currently in the admin pages, and the canonical fix.

Backgrounds

DO · Use #07090A on every page
Store Analytics
Every admin page sits on the same near-black surface. The eye reads it as one product.
DON'T · Cream parchment background
Store Analytics
The Analytics page is currently on a cream background. It breaks the system. Retire it.

Primary action button

DO · Emerald pill, dark text, soft glow
One primary per view. Secondary actions use the ghost pill — not a competing colour.
DON'T · Mix a teal-cyan gradient as a second primary
Currently visible on the Products / Orders / Customers pages. Two competing primary colours = no primary. Replace with .bs-btn-ghost.

Status badges

DO · One chip system, semantic colour roles
template.created Pending template.deleted
Same shape, same type, only colour role changes. Reads consistently across Audit, Orders, Products.

Three different visual systems for the same job. The user can't tell what's a "status" vs "category" vs "warning".

Migration checklist for Claude Code

  1. Theme. Force bg-bs-bg on every <body> / app shell. Remove the cream background from /admin/analytics.
  2. Page headers. Add the centered .bs-eyebrow + .bs-page-title + .bs-page-subtitle pattern to every page that lacks it (Customers is the worst offender).
  3. Buttons. Find every "Export records" / cyan-gradient button. Convert to .bs-btn-ghost. Keep .bs-btn-green for primary actions only.
  4. Chips. Replace solid-fill green/red audit pills, outlined order pills, and cream-fill product strain pills with .bs-chip-*. Map by role: success → green, pending → warn, processing → info, deleted/error → danger, category/role → gold.
  5. Stat cards. Standardise on .bs-stat. Eyebrow in mono, metric in Cormorant cream-gold. Apply on Analytics page especially.
  6. Tables. Wrap every list in .bs-table-wrap with the toolbar pattern (Cormorant title + count + search + filters). Numeric / ID / IP / timestamp cells in .mono.
  7. Cards. Replace any ad-hoc dark div containers with .bs-card.bs-card-pad. Card titles must be Cormorant 22px, never Inter.
  8. Hover. Add .bs-card's green-ring hover everywhere a card is interactive — but never on read-only summary cards.
  9. Audit. Grep the codebase for any # hex literal not in this token list — every one is a bug.

Hand this entire HTML file (or just § 07) to Claude Code along with the codebase. The tokens, components and migration checklist are self-contained — no external references required.
