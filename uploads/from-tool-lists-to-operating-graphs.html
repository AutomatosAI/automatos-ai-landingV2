For most of the last two years, tool use has meant: hand the model a list of functions and let it pick. That assumption is built into nearly every popular agent framework — LangChain, AutoGen, CrewAI, off-the-shelf function-calling APIs.

It's also the assumption that breaks first when you build a real AI operations platform.

This post is about what replaces it.

We've been building Automatos as a multi-tenant AI operations system: hundreds of internal capabilities, multiple specialist agents, a marketplace, layered memory, user-installed skills, workspace-scoped data, and a single orchestrator agent ("Auto") that has to coordinate across all of them. The naive tool-list pattern made it about ten percent of the way before falling over.

What follows is the architectural arc we've worked through, the research it connects to, and the direction we think serious agent systems will move in.

The shift in what "tool use" means

When tool calling was new, the question was whether models could reliably invoke functions, parse the output, and continue reasoning. That question is now mostly resolved. Frontier models call tools accurately when the toolset is small and well-described.

The unsolved problem is what happens when the toolset isn't small.

A working AI operations layer accumulates capability fast:

  • Core platform actions (analytics, agent management, memory operations, marketplace, monitoring)
  • Workspace tools (file system, code execution, deployment, browser, search)
  • User-installed integrations (CRM, calendar, email, custom APIs)
  • Marketplace skills, each with its own tool requirements
  • Specialist agents, each with their own assigned tools
  • Cross-cutting infrastructure (RAG, knowledge graph, vector search, document generation)

Anyone building seriously will hit a few hundred capabilities within months. The naive solution — list them all in the prompt and trust the model to pick — doesn't scale, and not for the reasons most people assume.

It's not primarily that the model can't find the right tool in a long list. It's that the model's effective attention is degraded by the irrelevant tools before tool selection even happens. The capability surface is wasting the attention budget the agent will need for actual reasoning.

This is the core engineering insight that has driven the last six months of architecture work: attention is a scarce resource, and tokens are how we ration it.

Attention budgets and the menu problem

A 200k-token context window is not 200k tokens of equally-attended reasoning surface. Long-context studies have shown for years that effective attention degrades with position and density. Stuff a critical tool definition in the middle of a 50k-token system prompt and the model may register that the tool exists without ever reaching for it during inference.

That gap — between available capability and active capability — is the real measure of agent intelligence in the operations regime.

The implication is uncomfortable for the standard agent-stack design: every additional tool in the prompt has a cost, even if it's never invoked. The cost is paid by all the other tools and reasoning the prompt was supposed to enable.

You don't escape this by enlarging the context window. Bigger windows let you put more in. They don't change the fact that putting more in dilutes attention to any specific part.

The cure has to come before the prompt is built.

The research backdrop

The academic community has been working this question from several angles, and the trajectory is clearer in retrospect than it felt at the time.

Toolformer (Schick et al., 2023) showed that language models can be fine-tuned to call APIs at the right moment and integrate results into subsequent reasoning. The contribution wasn't a tool-calling protocol; it was the demonstration that tool-use behavior can be learned by the model itself. paper

Gorilla (Patil et al., 2023) added retrieval over API documentation at inference time. This was the first major signal that tool selection — not tool invocation — was the limiting factor. Hallucinated API calls dropped sharply when the model was handed the right documentation rather than expected to memorize it.

ToolLLM (Qin et al., 2023) scaled the question to over 16,000 real-world APIs and built ToolEval as a benchmarking framework. The framework essentially formalized the move: at scale, retrieval-augmented tool calling is the only viable design. Memorization-based approaches don't compose.

A more recent thread is even more pointed.

ToolGen (Microsoft Research) argues that traditional approaches relying on tool descriptions in context are fundamentally limited by context length and retrieval overhead. The proposed direction is unified tool retrieval and generation, where tool selection is part of the model's generative process rather than an external preprocessing step. paper

SIT-Graph goes further. It represents tool-use experience as a graph: nodes are tools and states; edges are dependencies, sequences, and compact summaries drawn from prior agent trajectories. The agent traverses learned structure rather than reasoning over a flat list at every turn.

PlugMem applies an analogous argument to memory: raw experience trajectories are too verbose to retrieve over directly; the system needs to convert them into compact, knowledge-centric memory graphs. paper

Read together, the research is converging on a simple thesis: the next generation of agents won't survive on flat lists. They'll need learned structure across tools, skills, and memory.

The product question — what we've been working on — is how to bring that into a system that actually runs.

The Automatos architecture in layers

We worked the problem in layers, each one solving a specific constraint that surfaced when the previous one was in place.

Dynamic capability assignment

Originally, agent classes had hardcoded tool lists. A research agent shipped with research tools, a code agent with code tools, and so on. This worked at small scale but tied capability to identity in a way that prevented composition.

The first refactor moved tool granting to runtime. Agents request capabilities for a task; the orchestrator decides what to grant based on the task type, agent role, workspace permissions, and current context. Tool assignment became a runtime decision, not a class definition.

This was the first move from "agents own tools" to "the platform brokers tools."

A unified tool catalog

Once tools were brokered at runtime, the next problem was that tool definitions were still scattered across the codebase. Different consumers — agents, widgets, workflows, REST APIs — held their own copies of tool schemas. Drift was inevitable.

We introduced a centralized tool catalog with structured definitions: name, description, parameters, executor, credentials, access rules, intended consumers, audit metadata. Every consumer queries the catalog. The catalog is the source of truth.

This sounds bureaucratic, and it is, but the payoff is significant: anything you want to do across the platform's tool surface (analyze usage, restrict access, version-bump a schema, build a UI for tool management) requires one operation, not a dozen.

Platform self-management

Auto — the orchestrator agent — needed to operate the platform itself. Install skills, query analytics, manage agents, run missions, view memory. The naive approach would be to expose every internal API endpoint as a tool.

That's a category error.

Internal APIs are designed for service code, not for reasoning agents. They have field names that are meaningful to engineers, not to a model. They're often fine-grained where the agent needs coarse-grained operations. They expose the platform's implementation, not its behavior.

We introduced a curated Platform Action Registry: a layer of service-level operations that represent things the platform actually does, written in language that's meaningful for a model to reason over. Each action is an ActionDefinition — name, description, parameters with semantic descriptions, intent keywords, executor, permission requirements.

The agent uses the platform. It doesn't pretend to be the API.

Tool exposure as routing design

Even with a clean platform action layer, we hit the next tradeoff: should every action be a directly-callable tool schema, or should they sit behind a single dispatcher (platform_execute(action, params))?

A dispatcher saves token budget — one schema instead of fifty. But it adds a cognitive bridge: the model has to read action documentation, infer the right action name, and wrap it in the dispatcher call. Reliability suffers when the documentation is sparse or the action mapping is non-obvious.

The right answer turned out to be: both, deliberately. Some tools deserve to be directly callable schemas — common, high-trust, frequently invoked. Others can stay behind a dispatcher that resolves the action lazily from a documented catalog.

We promoted a small set of high-value actions to first-class schemas. Long-tail actions stayed behind the dispatcher. The decision criterion: frequency of use, criticality of failure, and how reliably the model could map intent to action.

Tool exposure is not a binary. It's a routing design with explicit tradeoffs per tool.

Configuration substrate stability

Routing intelligence rests on a stable foundation. If your orchestrator model, embedding model, memory service, RAG pipeline, and routing layer all have ambiguous or overlapping configuration, the agent becomes unpredictable before the prompt is even built.

We collapsed model and parameter configuration into three explicit tiers:

  • Auto — the orchestrator's model
  • System — background services (RAG, memory consolidation, document processing)
  • Embeddings — the embedding model used across all retrieval surfaces

Each tier has one source of truth, read through a single configuration module, with no scattered environment-variable reads or duplicated model selection logic. Configuration drift is now a build-time concern, not a runtime mystery.

This sounds like infrastructure plumbing because it is. Routing intelligence is impossible without it.

Prompt assembly as execution infrastructure

The hardest lesson came from a regression. Auto's behavior degraded over a couple of weeks through small, accumulated errors: stale settings cached in memory, duplicate identity blocks injected from two different places, a memory service URL malformed in a config refactor, skills truncated below their useful threshold, a legacy chat path still routing some queries through the wrong pipeline, broken tool-loop iteration limits.

Each error was small. Together, they made the agent behave like the wrong agent.

The fix wasn't a single bug. It was reframing the prompt as execution infrastructure — a deterministic build pipeline whose output is the program the model runs. If the wrong identity, the wrong skill, a stale memory, or a duplicated tool block enters that build, the system has a bug, even if no exception is raised.

We rebuilt prompt assembly as a sectioned pipeline with explicit inputs, deterministic ordering, and contract-style invariants per section. Identity goes here. Skills go there. Tools are constructed by this function. Memory is retrieved by that function. Each section has a token budget and a fallback strategy.

Once prompt assembly was treated as deterministic infrastructure, the regressions stopped being mysterious.

This is where we were standing when the real problem became visible.

The semantic action index

With everything else in place, an agent with roughly a hundred platform actions still has to read about all of them on every turn — unless we change what enters the prompt.

If the user asks "how's my website doing?" the prompt shouldn't allocate equal attention to agent deletion, marketplace installation, file operations, and infrastructure logs. It should rank actions semantically against the query and inject only the documentation likely to matter.

The mechanics:

  1. Embed each action at registration time. The embedding text is composed from the action's name, description, parameter descriptions, and intent keywords. Embeddings are computed using the configured embedding provider.
  2. Cache embeddings by content hash. If the action definition changes, the cache is invalidated for that action only. Cold-start cost is bounded; steady-state cost is near zero.
  3. At prompt-build time, embed the query (and a small recent-context window) using the same provider.
  4. Rank actions by cosine similarity. Apply permission filters first (an action the agent isn't allowed to invoke shouldn't influence ranking), then return the top-N.
  5. Inject a focused action summary into the platform-actions section of the prompt, instead of the full catalog.

The dispatcher tool (platform_execute) remains in the callable schema set. Capability is preserved end-to-end — the dispatcher can still execute any permitted action — but the prompt no longer has to describe every action every time.

This is a small change in code. The effect on attention budget is large.

We're calling the embedding/ranking layer the Action Semantic Index, and the prompt-construction integration point is the Platform Actions Section of the prompt assembly pipeline. The split matters: the index is reusable infrastructure; the section is one of several consumers.

Two levels of routing — and why the distinction matters

Semantic action ranking is deliberately not a redesign of tool calling. It separates two distinct levels:

  • Tool-schema routing — which callable tools the model receives in its function-calling schema.
  • Action-summary filtering — which platform actions are described inside the dispatcher's documentation when a dispatcher tool is present.

Conflating these is one of the major reasons agent stacks struggle to scale. A platform_execute schema and an ActionDefinition are not the same kind of object:

  • The schema tells the model that platform execution exists as a callable interface.
  • The action definitions tell the system which platform capabilities to describe in the prompt this turn.

Separating them lets the callable-schema set stay small, stable, and high-signal, while the described-capability set stays large, dynamic, and ranked per turn.

Most production agent systems we've looked at are conflating these and trying to optimize a single flat list. The optimization ceiling on a flat list is much lower than the ceiling on a properly separated two-level routing design.

The operating graph thesis

Semantic ranking is the floor, not the ceiling.

Cosine similarity over action embeddings will surface the right category of capability for a given query. It won't capture the structure of how an agent actually operates. That structure is sequential and relational.

A real agent doesn't pick one tool. It runs a sequence. An analyst usually queries analytics before generating a report. A marketer usually checks brand voice memory before drafting. A platform admin usually verifies marketplace state before installing. These are patterns, not single calls.

The longer-term direction we're moving toward is agent-specific operating graphs.

Conceptually:

  • Nodes are tools, actions, skills, and memory entries.
  • Edges encode learned relationships: used_after, requires, succeeds_for_intent, fails_for_intent, agent_prefers.
  • Edge weights are updated from successful and failed agent trajectories.
  • Per-agent subgraphs capture specialization — a research agent's graph is not a marketer agent's graph.

Semantic retrieval finds the entry points to the graph. The graph expands them into the likely chain.

The implementation is non-trivial. Storage choices include a property-graph database, a knowledge-graph extension to PostgreSQL, or a custom in-memory structure backed by JSON. Edge maintenance requires a feedback loop from agent execution telemetry, which we already collect for other reasons. The hard part is not storage; it's the design of the edge schema that captures genuinely useful operating patterns without growing into noise.

A tool list is what you have. An operating graph is what you know.

Skills and memory as parallel surfaces

Tools are not the only thing being naively dumped into prompts.

Skills — the reusable behavioral guides agents follow — face the same failure mode. A workspace might have 20 skills assigned to an agent. If every prompt loads all 20 in full, you're spending the same attention budget rendering instructions for skills the agent isn't going to use this turn.

The natural progression is mixed-density skill loading:

  • Full inclusion for the most relevant skills (top 1-2 by semantic relevance to the task).
  • Summary inclusion for the next tier (one or two lines per skill, enough for the model to know the skill exists and could be relevant).
  • Latent presence for the rest (the agent can request a skill by name; it loads on demand).
  • On-demand fetch for tasks that surface specific skills mid-execution.

This requires a skill semantic index analogous to the action semantic index, plus a load-policy decision per skill per turn. The infrastructure overlaps significantly with the action layer; the policy logic is distinct.

Memory is the third axis. A layered memory architecture with focus, working memory, short-term, long-term, and organizational tiers is only useful if the system decides which layer to retrieve from at each step. PlugMem makes a related argument from the academic side: convert raw agent experience into compact, knowledge-centric memory graphs so retrieval surfaces decision-relevant knowledge, not verbose trajectories.

We're treating tools, skills, and memory as three parallel surfaces with the same architectural pattern: a semantic index for retrieval, a learned graph for sequence/relational structure, and a per-turn load policy that respects context budget.

The pre-prompt intelligence layer as a product

So the direction isn't "put more in the prompt."

It's the opposite.

The direction is to build a pre-prompt intelligence layer that decides what deserves to enter the prompt at all.

That layer combines:

  • Semantic indexes over tools, actions, skills, and memories
  • Permission-aware filtering — surface only what the agent is allowed to use
  • Model-tier-aware routing — frontier models need less hand-holding than smaller ones; the prompt depth and skill density should adapt
  • Per-agent usage history — agents specialize, and their tool histories diverge
  • Graph edges learned from successful and failed tool sequences, with feedback from execution telemetry
  • Context budgets that treat attention as a scarce resource and enforce prompt size by section, with explicit fallback policies

None of these threads are individually new. API retrieval exists. Tool learning exists. Tool graphs exist. Memory graphs exist. Trajectory reuse exists. What's largely missing in production agent stacks is the product architecture that combines them into a coherent operations system.

This is where the next round of agent capability will come from. Not bigger models. Not bigger context windows. Not more tools. A smarter pre-prompt layer that gives the model the right slice of the operating system at the moment.

Open problems

A few things are still genuinely unresolved, and we'll write about them as we work through them:

  1. How aggressive should the pruning be? Surfacing too few actions risks missing a needed one and forcing fallback discovery loops. Surfacing too many wastes attention. The right threshold is task-dependent and probably model-dependent.
  2. How does the operating graph initialize on a new agent? Cold-start agents have no execution history. Bootstrapping from analogous agents' graphs (or from a generic prior) is a design choice with real consequences for early-life behavior.
  3. How do you prevent the operating graph from ossifying? A graph that strongly weights past patterns can prevent agents from discovering new ones. Some exploration policy is required.
  4. How do you handle multi-turn drift? A query may start in one capability domain and migrate. Re-ranking on every turn is expensive; not re-ranking is wrong.
  5. How do you design edges that survive schema evolution? When an action gets renamed or replaced, the graph edges referencing it need a migration story. Otherwise the graph rots.

These are the kinds of problems that don't get attention in agent demos and turn out to dominate the engineering of real systems.

Closing

An AI operating system should not ask the model to remember the whole operating system.

It should give the model the right part of the operating system at the moment.

Tool lists got us to chatbots. They won't get us to agents that run businesses.

What comes next is a layer that thinks about the prompt before the prompt thinks. A layer that ranks, prunes, expands, and orders. A layer that learns from every successful and failed turn what to surface for the next one.

That's the build. We'll write the next post when the operating graph layer is in production.
