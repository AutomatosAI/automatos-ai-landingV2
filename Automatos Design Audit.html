Executive Summary

Overall Assessment

Automatos AI has a strong design system foundation (comprehensive DESIGN_SYSTEM.md, shared components, token architecture). However, implementation drift across pages creates inconsistency. The platform is functional and the visual identity is clear, but tightening enforcement will make it feel more cohesive and premium.

  • Good: CSS variable token system is in place and mostly followed
  • Good: Glass morphism + orange glow signature is distinctive
  • Issue: Modals have inconsistent styling (some missing card-glow, some using hardcoded styles)
  • Issue: Button hierarchy violations (multiple primary CTAs per view)
  • Issue: Mixed border radius usage (some forms use rounded-lg instead of rounded-2xl)
  • Issue: Profile page bypasses design system entirely (hardcoded Tailwind colors, custom layout)
  • Issue: Wizard flows don't match ItemCard or standard card patterns

02

Profile Page Bypasses Design System

🚨

Hardcoded Colors & Custom Layout

Severity: Critical • Location: settings/profile/page.tsx
Token System Violation

Problem: The profile page (settings/profile/page.tsx) is the only page in the app that doesn't use PageHeader or shared components. It uses:

  • Hardcoded Tailwind colors (text-white, text-slate-400, bg-slate-800/50)
  • Custom heading layout instead of PageHeader
  • Direct gradient classes (from-orange-500 to-purple-600) instead of using gradient-text
  • Inconsistent input styling (rounded-lg instead of rounded-2xl)

Impact: The profile page feels like it belongs to a different app. It doesn't match the rest of Automatos.

// Current (Hardcoded)
<h1 className="text-3xl font-bold text-white mb-2">
  Profile Settings
</h1>

// Should be (Using PageHeader)
<PageHeader
  title="Profile"
  titleAccent="Settings"
  subtitle="Manage your account information and preferences"
/>
Refactor Required
  • Replace custom header with PageHeader
  • Replace all hardcoded colors with CSS variable classes (text-foreground, text-muted-foreground, bg-secondary)
  • Change input rounded-lg → rounded-2xl
  • Use glass-card class for the main card
  • Replace gradient button classes with standard Button component
03

Button Hierarchy Violations

⚡

Multiple Primary CTAs Per View

Severity: Major • Impact: User Focus
Design System §7

Problem: The design system states: "Maximum ONE solid orange (primary) button per view/section." However, several pages have multiple primary buttons competing for attention.

Examples:

  • Publish Wizard: "Save Draft" and "Next" both appear as primary in some steps
  • Profile Page: "Edit Profile" and "Change Avatar" both use primary gradient styling
  • Modal Footers: Some have both "Save" and "Submit" as primary

Why it matters: Multiple primary buttons create decision paralysis. Users don't know which action is most important.

❌ Current (Confusing)

Both buttons look equally important. Which should I click?

✅ Correct (Clear)

"Next" is clearly the primary action. "Save Draft" is available but de-emphasized.

Button Hierarchy Rules
  • Primary (solid orange): ONE per view — the main action users should take
  • Outline (orange border): Secondary actions — "Details", "Export", "Refresh"
  • Ghost (subtle): Tertiary actions — "Cancel", "Save Draft", icon buttons
  • Destructive (red): Dangerous actions only — "Delete", "Remove"
04

Border Radius Inconsistency

📐

Mixed Radius Values Across Forms

Severity: Major • Impact: Visual Consistency
Design System §5

Problem: The design system mandates rounded-2xl (1rem / 16px) for inputs, cards, and buttons. However, several pages use rounded-lg (0.5rem / 8px) instead.

Found in:

  • Profile page form inputs
  • Publish wizard text inputs
  • Some modal input fields

Impact: The platform looks inconsistent. Some pages feel more rounded and friendly; others feel tighter and more conservative. This undermines the visual identity.

// Wrong
<Input className="rounded-lg" />

// Correct (already default in Input component)
<Input />  // Uses rounded-2xl by default
Border Radius Reference
  • rounded-full (pill): Badges, pills, search bars, select triggers, tabs
  • rounded-2xl (1rem): Buttons, cards, modals, inputs, textareas — THE DEFAULT
  • rounded-xl (0.75rem): Dropdown menu content panels
  • rounded-lg (0.5rem): Dropdown menu items, small controls
  • Never use: rounded-sm, rounded-md, or sharp corners

Wizard Cards Don't Match ItemCard

🎨

Custom Card Layout in Publish Wizard

Severity: Minor • Location: marketplace/publish/page.tsx
Component Library

Observation: The publish wizard uses raw Card + CardHeader + CardContent instead of the shared ItemCard component. While not technically wrong, it means the wizard cards don't benefit from updates to ItemCard (like consistent animations, hover states, or structural improvements).

Recommendation: Consider whether wizard steps could use ItemCard or a variant. If not, document why wizards are an exception.

07

Featured Widget Cards Have Custom Structure

🌟

Horizontal Scroll Cards Bypass ItemCard

Severity: Minor • Location: marketplace/widgets/page.tsx
Component Library

Observation: The featured widgets section hand-builds cards with Card + custom layout instead of using ItemCard. The structure is similar but not identical.

Trade-off: Custom layout allows for the specific horizontal scroll pattern, but it means featured cards and grid cards don't share structure. If ItemCard gets an improvement (like better badge positioning), featured cards miss it.

Recommendation: Either extend ItemCard to support a "compact horizontal" variant, or document that featured carousels are an intentional exception.

08

Stat Card Icon Colors Are Inconsistent

📊

Mixed Semantic Color Usage

Severity: Minor • Impact: Color Meaning
Color System

Observation: The StatsBar component on the Agent Management page uses a mix of semantic color tokens:

  • "Total Agents" → text-primary (orange) ✅
  • "Active Agents" → text-[hsl(var(--success))] (green) ✅
  • "Needs Attention" → text-destructive (red) ✅
  • "Avg Success Rate" → text-[hsl(var(--agent))] (purple) ❓

Question: What does purple (--agent) mean? It's used for "Avg Success Rate" but the DESIGN_SYSTEM.md doesn't specify when to use the agent color vs. primary vs. info. This makes it hard for other developers to choose the right color.

Recommendation: Document Semantic Color Meanings
  • Add a "Status Mapping Reference" table to DESIGN_SYSTEM.md §14
  • Define what --agent (purple) is for: "Agent-specific metrics like performance, capability scores"
  • Define when to use --info (blue) vs. --agent (purple) for metrics
  • Provide examples for each semantic color in context

What's Working Well

Strengths to Preserve
  • Shared component adoption is high: PageHeader, StatsBar, FilterTabs are widely used across Agent Management, Marketplace, and other pages
  • Glass morphism signature is distinctive: The backdrop-blur + orange glow creates a premium, futuristic feel that differentiates Automatos from generic dashboards
  • Token architecture is clean: CSS variables are well-organized, HSL format enables alpha manipulation, and the dark/light/matte theme variants work
  • Documentation is comprehensive: DESIGN_SYSTEM.md is one of the most detailed design docs I've seen (32KB!). Most platforms don't have 1/10th of that
  • Semantic colors are well-defined: Success/warning/destructive/info/agent tokens exist and are mostly used correctly
  • The design system exists and is *mostly* followed: ~75% adherence is actually good for a fast-moving startup. The foundation is solid; just needs tightening
🎯

Final Verdict

Is Automatos AI's UI/UX good?

Yes, with caveats. The platform has a strong visual identity (glass + orange glow), a comprehensive design system, and good UX fundamentals (clear hierarchy, responsive layouts, accessible controls). The shared component library shows architectural discipline.

However, implementation drift creates inconsistency. Some pages feel polished and cohesive (Agent Management, Marketplace grid); others feel rushed or hand-coded (Profile page, some modals). The design system is excellent — the problem is enforcement.

Quick Wins (4-8 hours of work)
  • Refactor Profile page to use design system components
  • Add card-glow to all modals
  • Fix button hierarchy violations (demote secondary CTAs)
  • Replace custom search bars with SearchInput

Result: The platform will feel 30-40% more polished and consistent with minimal effort.
